  <files>proxy.ts, middleware.ts</files>
  <action>
    Perform the rename and rewrite as a single atomic operation:

    1. Delete middleware.ts (it will be replaced by proxy.ts ŌĆö the old file must not remain or Next.js will error with two proxy/middleware exports).

    2. Create proxy.ts at the project root with these exact characteristics:

    Imports:
    - `NextResponse` and `NextRequest` from 'next/server'
    - `getSessionCookie` from 'better-auth/cookies'
    - `checkRateLimit` from '@/lib/ratelimit' (preserving existing import)

    Export the function named `proxy` (NOT `middleware` ŌĆö Next.js 16 breaking change per RESEARCH.md Pitfall 4). The function must:

    a. Apply the existing rate-limit block: if pathname starts with '/api', run `checkRateLimit(ip)` exactly as the old middleware.ts did (X-Forwarded-For, fallback 'local-dev', 429 response with same headers, `X-RateLimit-Remaining` on success). Do not change any rate-limit logic.

    b. After the rate-limit block (or if not an API path), add the session-cookie guard:
    - Call `getSessionCookie(request)` ŌĆö this returns the cookie string or null, no DB call.
    - If `request.nextUrl.pathname` starts with '/history' and sessionCookie is falsy, return `NextResponse.redirect(new URL('/sign-in', request.url))`.
    - If `request.nextUrl.pathname` starts with '/sign-in' and sessionCookie is truthy, return `NextResponse.redirect(new URL('/', request.url))`.
    - Otherwise return `NextResponse.next()`.

    Export `config` matcher to cover both API routes (for rate limiting) and page routes (for session guard), excluding Next.js internals and static assets:
    ```
    matcher: ['/((?!_next/static|_next/image|favicon.ico).*)']
    ```
    This replaces the old `'/api/:path*'` matcher ŌĆö the rate-limit block still only runs when pathname starts with '/api', so the behaviour is unchanged for API routes.

    CRITICAL: Do NOT import `cookies` from `next/headers` in proxy.ts ŌĆö that is a Server Component API and will error in the proxy context. `getSessionCookie(request)` uses the request object directly.

    CRITICAL: The rate-limit check includes an `await` on `checkRateLimit` ŌĆö the proxy function must remain `async`.
  </action>

    Create all four files. Each file is small and self-contained.

    --- src/lib/auth.ts ---

    Server-only better-auth configuration using the memory adapter for Phase 4. Exact shape:

    - Import `betterAuth` from 'better-auth'
    - Import `memoryAdapter` from 'better-auth/adapters/memory'
    - Import `nextCookies` from 'better-auth/next-js'

    Declare `memoryDB` as a `Record<string, any[]>` with four keys: `user`, `session`, `account`, `verification` ŌĆö each an empty array. This shape is required by the memory adapter.

    Call `betterAuth({ ... })` with:
    - `baseURL: process.env.BETTER_AUTH_URL`
    - `secret: process.env.BETTER_AUTH_SECRET`
    - `database: memoryAdapter(memoryDB)`
    - `emailAndPassword: { enabled: true }` ŌĆö built-in feature, no plugin import needed
    - `socialProviders: { google: { clientId: process.env.GOOGLE_CLIENT_ID as string, clientSecret: process.env.GOOGLE_CLIENT_SECRET as string } }`
    - `session: { expiresIn: 60 * 60 * 24 * 7, updateAge: 60 * 60 * 24 }` ŌĆö 7-day session, refresh if older than 1 day
    - `plugins: [nextCookies()]` ŌĆö MUST be the last entry in the array; fixes Set-Cookie in Server Actions

    Export as `export const auth = betterAuth({ ... })`.

    NOTE: Phase 5 will replace `memoryAdapter(memoryDB)` with `drizzleAdapter(db, { provider: 'pg' })` ŌĆö the rest of the config stays identical. Do NOT add a comment about this in the file (keep it clean).

    --- src/lib/auth-client.ts ---

    Browser-side auth client. Exact shape:

    - Import `createAuthClient` from 'better-auth/react'
    - Call `createAuthClient({ baseURL: process.env.NEXT_PUBLIC_APP_URL ?? 'http://localhost:3000' })`
    - Export as `export const authClient`
    - Re-export named: `export const { signIn, signOut, signUp, useSession } = authClient`

    This file is imported by Client Components. It must NOT import from 'server-only'.

    --- src/lib/dal.ts ---

    Server-only Data Access Layer for session verification. Exact shape:

    - First line: `import 'server-only'`
    - Import `auth` from '@/lib/auth'
    - Import `headers` from 'next/headers'
    - Import `redirect` from 'next/navigation'
    - Import `cache` from 'react'

    Export `verifySession` wrapped with `cache()`:
    ```
    export const verifySession = cache(async () => {
      const session = await auth.api.getSession({ headers: await headers() })
      if (!session) redirect('/sign-in')
      return { isAuth: true as const, userId: session.user.id, user: session.user }
    })
    ```

    The `cache()` wrapper deduplicates the DB lookup when `verifySession()` is called multiple times within the same server render pass (RSC + route handler both calling it).

    CRITICAL: This file calls `redirect()` on missing session. In SSE route handlers, `redirect()` would break the stream ŌĆö Plan 04-03 handles SSE routes separately by catching the redirect and returning a 401 instead.

    --- src/app/api/auth/[...all]/route.ts ---

    Create the directory `src/app/api/auth/[...all]/` and the route file. Exact content:

    - Import `auth` from '@/lib/auth'
    - Import `toNextJsHandler` from 'better-auth/next-js'
    - `export const { GET, POST } = toNextJsHandler(auth)`

    That is the complete file. `toNextJsHandler` returns `{ GET, POST, PATCH, PUT, DELETE }` ŌĆö only GET and POST are needed for the standard email/Google flow.

    Do NOT add `export const dynamic = 'force-dynamic'` to this file ŌĆö better-auth handles caching internally.
