```
    try {
      const result = await callWithRetry(userPrompt, (text) => {
        controller.enqueue(sseEvent({ type: 'chunk', content: text }))
      })
      controller.enqueue(sseEvent({ type: 'result', data: result }))

      // Merge interview data into the existing application record — non-fatal if missing/failed

        try {
          await updateInterviewData(userId, parsed.data.applicationId, result)

          // The Q+A result is already in the client state; only history persistence is affected
        }
      }
    } catch (err) {
      const message = err instanceof Error ? err.message : 'Question generation failed'
      controller.enqueue(sseEvent({ type: 'error', message }))
    } finally {
      controller.enqueue(sseEvent({ type: 'done' }))
      controller.close()
    }
```

Key constraints:
- The `updateInterviewData` call is INSIDE the outer try block (after `result` is emitted) but inside its OWN nested try/catch, so a DB failure does not trigger the outer catch's `error` event.
- The `if (parsed.data.applicationId)` guard means no DB call when applicationId is absent — no DAL call, no error.
- `result` is already emitted to the client before the merge attempt — the user has their Q+A data regardless of whether the merge succeeds.
- `done` fires in `finally` as before — always.
- `updateInterviewData` is called ONLY after the full `result` is available (after `callWithRetry` returns), never during streaming — per the requirement that `interviewData` must be a complete Q+A object.

Do not change: imports beyond the dal line, `callWithRetry`, response headers, or `parsed.data.resumeText`/`jdText` usage.
