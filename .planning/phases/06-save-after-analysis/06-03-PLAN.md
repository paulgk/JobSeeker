**Change 2 — Inject save logic into the ReadableStream.start() try block.**

Current try block inside `start(controller)`:
```

      const result = await callWithRetry(userPrompt, (text) => {
        controller.enqueue(sseEvent({ type: 'chunk', content: text }))
      })
      controller.enqueue(sseEvent({ type: 'result', data: result }))
```

Replace with:
```
    try {
      const result = await callWithRetry(userPrompt, (text) => {
        controller.enqueue(sseEvent({ type: 'chunk', content: text }))
      })

      // Attempt to save — nested try/catch so save failure is non-fatal (per D-06)
      let applicationId: string | undefined
      try {
        const { jobTitle, company } = await extractJobMeta(parsed.data.jdText)
        applicationId = await saveApplication(userId, {
          jobTitle,
          company,
          resumeText: parsed.data.resumeText,
          jdText: parsed.data.jdText,
          matchScore: result.overallScore,
          analysisData: result,
        })
      } catch (saveErr) {
        const saveMessage = saveErr instanceof Error ? saveErr.message : 'Could not save to history'
        controller.enqueue(sseEvent({ type: 'save_error', message: saveMessage }))
      }

      // Emit result with applicationId (undefined if save failed — client handles null case)
      controller.enqueue(sseEvent({ type: 'result', data: result, applicationId }))
    } catch (err) {
      const message = err instanceof Error ? err.message : 'Analysis failed'
      controller.enqueue(sseEvent({ type: 'error', message }))
    } finally {
      controller.enqueue(sseEvent({ type: 'done' }))
      controller.close()
    }
```

Critical ordering constraints:
- `save_error` (if emitted) fires BEFORE `result` — client receives the banner notice before the result state, so the UI can show the toast alongside the result rather than after.
- `result` fires BEFORE `done` — the finally block ensures `done` always fires last.
- `callWithRetry` failure path is unchanged — analysis failures still emit `error` (fatal), not `save_error`.

PITFALL — double-save on "Run again": Each call to `start()` creates a new INSERT row. This is intentional (each analysis run is a distinct record). The `savedApplicationId` stale-state bug (stale ID being passed to interview prep when user clicks "Run again") is handled in plan 06-04 by resetting `savedApplicationId` in page.tsx when a new analysis starts.

Do not change: the `callWithRetry` function, the response headers, the `dynamic`/`runtime`/`maxDuration` exports, or the request body parsing block.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM result → DB write | Result data (AnalysisResult) written verbatim to JSONB; no user-controlled SQL injection surface via Drizzle ORM |
| save failure → client event | save_error event message is internal error text, not user-supplied data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-06 | Information Disclosure | save_error message sent to client | accept | Message is internal error text from Postgres/Drizzle; may expose DB error strings. Acceptable for MVP — user is authenticated and the info helps diagnose save failures |
| T-06-07 | Tampering | Double-save via rapid concurrent requests | accept | Each save is an INSERT; duplicate rows are benign extra entries in history. Not a security issue (only the authenticated user sees their own rows). Idempotency deferred to Phase 7. |
| T-06-08 | Denial of Service | extractJobMeta() Haiku call adds ~300ms to each analyse request | accept | Adds latency before result emission but within the 60s maxDuration window; cost is ~$0.0002/call |
</threat_model>
