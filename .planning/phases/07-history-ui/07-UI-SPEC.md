## Color

All tokens from `globals.css`. Phase 7 is dark-first — dark mode values govern the visual hierarchy.

| Role | Dark Mode Value | Usage |
|------|----------------|-------|
| Dominant (60%) | `--background: oklch(0.145 0 0)` | Page background, /history list page, /history/[id] page |
| Secondary (30%) | `--card: oklch(0.205 0 0)` | Application row cards, detail page panels, AuthHeader background |
| Accent (10%) | `--primary: oklch(0.922 0 0)` | Re-run button, active status badge (Saved), focus rings |
| Destructive | `--destructive: oklch(0.704 0.191 22.216)` | Rejected status badge only |

Accent reserved for:
1. "Re-run Analysis" primary button (Button variant="default")
2. "Saved" status badge (first/default state — primary variant)
3. Focus rings (ring-ring via shadcn default)

Status badge color mapping (using existing Badge variants):
- **Saved** → `variant="default"` (primary/white on dark)
- **Applied** → `variant="secondary"` (muted surface)
- **Interviewing** → `variant="secondary"` with `--chart-1` accent text (`oklch(0.87 0 0)`)
- **Offer** → `variant="default"` with success-implied styling (keep primary, no new token)
- **Rejected** → `variant="destructive"` (destructive tint)

Note: No new color tokens are introduced. All status states map to existing badge variants.

Source: `globals.css` dark values, `badge.tsx` variant definitions, `action-list.tsx` impact badge pattern.

## Layout Contracts

### AuthHeader (new, shared across all authenticated pages)

```
[JobSeeker wordmark]                    [user email — muted]  [Sign out — ghost button]
```

- Height: 64px (h-16)
- Background: `bg-card` (secondary surface)
- Bottom border: `border-b border-border`
- Navigation: includes /history link (History) with active state underline
- Positioned: sticky top-0 z-10 — stays visible during scroll
- Sign out calls better-auth signOut(), redirects to /sign-in

### /history List Page

```
AuthHeader
─────────────────────────────────────────────────────
h1: "My Applications"
─────────────────────────────────────────────────────
[Company name — label]  [Job title — body]  [Score badge]  [Status select]  [Date — muted]
[Company name — label]  [Job title — body]  [Score badge]  [Status select]  [Date — muted]
...
```

- Each row: `bg-card rounded-2xl ring-1 ring-border px-6 py-4` — matches home page card pattern
- Row is clickable → navigates to /history/[id]
- Status select within row: inline, 140px wide, does NOT navigate on change
- Score badge: number + "%" in `text-sm font-semibold`, colored with `--chart-1` when score ≥ 70 (matches ScoreCard pattern)
- Date: `text-xs text-muted-foreground` formatted as "May 26, 2026"
- Row list: `space-y-3` vertical gap
- Page max-width: `max-w-4xl mx-auto px-6 py-8` — narrower than home (list benefits from focused column)

**Focal Point:** The h1 "My Applications" headline draws the eye first; the application card list below it is the primary visual anchor. Users scan vertically down the card stack — company name (label weight) acts as the leftmost anchor per row.

### /history/[id] Detail Page

```
AuthHeader
─────────────────────────────────────────────────────
← Back to History
Company name / Job title                [Status select]  [Re-run Analysis button]
─────────────────────────────────────────────────────
[Tabs: Analysis | Interview Prep]
  Analysis tab:
    ScoreCardDisplay
    ─────────────
    ActionListDisplay
    ─────────────
    KeywordBadgesDisplay
    ─────────────
    RewriteDiffReadOnly (accepted rewrites only, no action buttons)

  Interview Prep tab:
    QuestionCardDisplay list (when interviewData exists)
    Empty state copy (when interviewData is null)
```

- Back link: `← Back to History` as an `<a>` or Next.js `<Link>`, `text-sm text-muted-foreground hover:text-foreground`
- Detail page max-width: `max-w-3xl mx-auto px-6 py-8`
- Tabs default to "Analysis" tab
- "Interview Prep" tab shows only when `interviewData` is non-null on the application record; otherwise show tab with empty state (not hidden — HIST-04 requires it to be visible)
- Re-run button: navigates to `/?applicationId={id}` — page.tsx reads query param, pre-populates resumeText + jdText from DB, sets `analysing=false` so user can confirm before running

**Focal Point:** The company name / job title heading (Display 24px, semibold) is the visual anchor. The "Re-run Analysis" button (accent color, right-aligned in the header row) is the primary interactive anchor — it is the only accent-colored element on the page above the fold.

## Interaction Contracts

### Status Update (HIST-02)

- Trigger: User changes Select value on list row or detail page header
- Behavior: Optimistic UI update — badge updates immediately in local state; PATCH `/api/applications/[id]/status` fires in background
- On PATCH success: No visible change (already updated)
- On PATCH failure: Revert to previous badge value; show inline error text below the select: "Couldn't save status. Try again."
- Select width: 140px fixed; does not expand the row height

### Re-run Navigation (HIST-05)

- Trigger: "Re-run Analysis" button click on detail page
- Behavior: `router.push('/?applicationId=' + id)` — no pre-fetch needed; page.tsx reads `searchParams.applicationId`, calls `getApplicationById(id)` for resumeText + jdText, pre-populates both panels
- Loading state: Button shows "Loading…" with disabled state during the navigation + data fetch
- No confirmation required — re-run is non-destructive (creates a new application record)

### History List Load (HIST-01)

- Trigger: Page mount
- Behavior: Server component — data fetched server-side via `getApplications(userId)` DAL call; no loading skeleton needed for initial load
- Empty state: Rendered server-side when array length === 0
